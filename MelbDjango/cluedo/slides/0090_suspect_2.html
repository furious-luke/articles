This is the beginning of the Task serializer definition. You can see the list of related serializers that will be used when including certain relationships in the resultant JSON when querying the API. Take note that they're all defined as strings. 


from django.utils.module_loading import import_string as import_class_from_dotted_path

def get_included_serializers(serializer):
    included_serializers = copy.copy(getattr(serializer, 'included_serializers', dict()))

    for name, value in six.iteritems(included_serializers):
        if not isinstance(value, type):
            if value == 'self':
                included_serializers[name] = (
                    serializer if isinstance(serializer, type) else serializer.__class__
                )
            else:
                included_serializers[name] = import_class_from_dotted_path(value)

    return included_serializers

Now let's look at the DRF-JSONAPI function that looks up these serializers. This is run every time a serializer is constructed. Notie the call near the bottom to "import_class_from_dotted_path"; that's going to be called on every one of the string paths we saw just before. You'd think that once a class has already been imported it would keep a lookup in a table for it, but seemingly that's not the case, as it's still consuming about half a second in total.

So how can we fix this? Well, the easiest way was to add a table to the code to keep each serializer class after we've looked it up once. The next time we grab it straight from the table.
